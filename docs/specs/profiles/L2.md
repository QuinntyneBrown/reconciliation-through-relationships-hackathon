# Participant Profiles (PROF) — L2 Detailed Requirements

Each requirement traces to exactly one L1 in [L1.md](L1.md). Acceptance criteria are written so a test can be authored without further clarification; user-visible strings are quoted verbatim from source. Requirements state intended behavior; a `**Status:** Not satisfied` line marks requirements the running system does not yet meet (see [known gaps](../GAPS.md)). Conventions, the ID scheme, and the acceptance-test trace-header format are defined in the [specification registry](../README.md).

---

## L2-PROF-031: Profile view

**Traces to:** L1-PROF-008

`/profile/[userId]` shall render the participant's shareable profile.

**Acceptance Criteria:**

1. Given an existing participant, when their profile loads, then it shows avatar or initials, name and age, Indigenous or Non-Indigenous badge, participation categories, city and province, treaty area (when present), bio under "About", faith tradition (with the self-described text when "other"), languages, preferred formats, and interests.
2. Given a non-existent user id, when the profile is requested, then the page responds not-found.

---

## L2-PROF-032: Facilitator profiles are not viewable

**Traces to:** L1-PROF-008

Facilitator accounts shall not be exposed through the profile page.

**Acceptance Criteria:**

1. Given a facilitator's user id, when `/profile/{id}` is requested by a participant, then the page responds not-found.

---

## L2-PROF-033: Own-profile shortcut

**Traces to:** L1-PROF-008

`/profile/me` shall resolve to the signed-in user's profile.

**Acceptance Criteria:**

1. Given a signed-in participant, when they request `/profile/me`, then they are redirected to `/profile/{their-id}` and see their own profile without a connect action.

## L2-PROF-034: Profile connect action state machine

**Traces to:** L1-PROF-008

The profile page's connect action shall reflect the current connection state and drive its transitions.

**Acceptance Criteria:**

1. Given no connection with the profile's owner, when the viewer activates "Connect with {name}", then a `connections` row is created (`participant_a` = viewer, connected; `participant_b` = owner, not connected; no match required), a `connect_request` notification is created for the owner, and the toast "Connection request sent to {name}!" appears.
2. Given the viewer already sent a request, when the profile renders, then the action shows a disabled "Request pending" plus a "Cancel" action; cancelling deletes the connection and shows "Connection request cancelled."
3. Given the owner sent the viewer a request, when the profile renders, then the action is "Respond to request" linking to `/connections/{id}`.
4. Given an active connection, when the profile renders, then the action is "Open chat" linking to `/connections/{id}`.
5. Given the connection insert fails, when the viewer tries to connect, then the toast "Couldn't send your connection request. Please try again." appears and no success toast is shown.
