# Security & Data Foundations (FOUND) — L2 Detailed Requirements

Each requirement traces to exactly one L1 in [L1.md](L1.md). Acceptance criteria are written so a test can be authored without further clarification; user-visible strings are quoted verbatim from source. Requirements state intended behavior; a `**Status:** Not satisfied` line marks requirements the running system does not yet meet (see [known gaps](../GAPS.md)). Conventions, the ID scheme, and the acceptance-test trace-header format are defined in the [capability registry](../README.md).

---

## L2-FOUND-063: Row-level security on every table

**Traces to:** L1-FOUND-015

All eleven application tables shall have row-level security enabled with least-privilege policies.

**Acceptance Criteria:**

1. Given the migrated schema, when policies are inspected, then RLS is enabled on `profiles`, `learning_modules`, `learning_progress`, `matches`, `connections`, `messages`, `meetings`, `cohorts`, `cohort_members`, `system_settings`, and `notifications`.
2. Given participant-owned rows (`learning_progress`, own `profiles` row), when accessed, then only the owner (and facilitators, where read access is intended) can read or write them.

## L2-FOUND-064: Profile data requires authentication

**Traces to:** L1-FOUND-015
**Status:** Not satisfied — the `profiles` SELECT policy's `(learning_completed AND onboarding_completed)` branch has no authentication requirement, and blanket grants expose onboarded profiles (including coordinates and personal boundaries) to the `anon` role (GAP-003).

Profile rows shall be readable only by authenticated users: the owner, facilitators, and — for participants who completed the journey — other authenticated participants.

**Acceptance Criteria:**

1. Given an unauthenticated request with only the public anon key, when it selects from `profiles`, then no rows are returned.
2. Given an authenticated participant, when they select from `profiles`, then they see their own row plus rows of participants with both journey flags complete.

## L2-FOUND-065: Map consent enforced server-side

**Traces to:** L1-FOUND-015
**Status:** Not satisfied — consent is filtered only in the browser; non-consenting participants' coordinates are shipped to the client (GAP-004).

Participants who have not consented to map display shall never have their coordinates delivered to another user's client.

**Acceptance Criteria:**

1. Given a participant with `map_consent = false`, when another participant loads any map surface, then that participant's `lat`/`lng` are absent from every payload sent to the browser, not merely hidden by client-side filtering.

## L2-FOUND-066: Notification creation is restricted to legitimate actors

**Traces to:** L1-FOUND-015
**Status:** Not satisfied — the INSERT policy allows any authenticated user to create a notification for any user with arbitrary content (GAP-005).

Notifications shall only be creatable in the context of the event they describe, preventing spoofed or spam notifications.

**Acceptance Criteria:**

1. Given an authenticated user with no relationship to the target user, when they attempt to insert a notification addressed to that user, then the write is rejected.
2. Given the legitimate flows (connection request, match approval, new message, meeting scheduled), when they run, then their notifications are still created.

---

## L2-FOUND-067: Message and meeting access limited to connection parties

**Traces to:** L1-FOUND-015

Messages and meetings shall be readable and writable only by the connection's participants and facilitators.

**Acceptance Criteria:**

1. Given a participant who is not a party to a connection, when they attempt to select or insert `messages` or `meetings` rows for it, then the database policy rejects the operation.
2. Given a connection party or a facilitator, when they access those rows, then the operation succeeds.

---

## L2-FOUND-068: Secrets remain server-side

**Traces to:** L1-FOUND-015

High-privilege credentials shall never reach the browser.

**Acceptance Criteria:**

1. Given the application source under `src/`, when it is inspected, then the Zoom client secret and account id are read only inside the server route `/api/zoom/create-meeting`, and the Supabase service-role key is never referenced (it is used only by seed and test tooling).
2. Given browser-delivered code, when it is inspected, then the only Supabase credential present is the public anon key.

---

## L2-FOUND-069: User content sanitized before HTML rendering

**Traces to:** L1-FOUND-015
**Status:** Not satisfied — the participant map popup injects unescaped profile fields via `setHTML` (GAP-007).

User-provided profile fields shall be escaped or sanitized wherever they are rendered as HTML rather than text.

**Acceptance Criteria:**

1. Given a profile whose first name contains markup such as `<img onerror=…>`, when that participant appears in a map popup, then the markup is rendered inert as text and no script executes.

## L2-FOUND-070: Status vocabularies match application writes

**Traces to:** L1-FOUND-016
**Status:** Not satisfied — the `connections.status` CHECK permits only `pending` and `active`; the application also writes `pending_review` (GAP-001).

Every status value the application writes shall be permitted by the corresponding CHECK constraint.

**Acceptance Criteria:**

1. Given the mutual-consent flow with auto-matching disabled, when the application sets `connections.status = 'pending_review'`, then the database accepts the write.
2. Given the migrated schema, when each CHECK-constrained column is compared with the values the application writes (`profiles.sex`, `profiles.faith_tradition`, `profiles.role`, `matches.status`, `connections.status`, `notifications.type`, `learning_modules.content_type`/`audience`), then every written value is permitted.

## L2-FOUND-071: Database triggers maintain rows

**Traces to:** L1-FOUND-016

Signup and update triggers shall maintain profile rows automatically.

**Acceptance Criteria:**

1. Given a new `auth.users` row, when it is inserted, then the `handle_new_user` trigger creates a matching `profiles` row with role `participant`.
2. Given any `profiles` update, when it commits, then `updated_at` is set to the update time by the `handle_updated_at` trigger.

---

## L2-FOUND-072: Realtime publication covers subscribed tables

**Traces to:** L1-FOUND-016
**Status:** Not satisfied — the UI subscribes to `connections` updates, but only `messages` and `notifications` are in the publication (GAP-002).

The `supabase_realtime` publication shall include every table the UI subscribes to.

**Acceptance Criteria:**

1. Given the chat page open on a pending connection, when the connection's status changes to `active` in the database, then the open page updates live (and shows "You're now connected! You can start chatting.") without a reload.
2. Given the publication, when it is inspected, then `messages`, `notifications`, and `connections` are all published.
