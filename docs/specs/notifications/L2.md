# Notifications (NOTIF) — L2 Detailed Requirements

Each requirement traces to exactly one L1 in [L1.md](L1.md). Acceptance criteria are written so a test can be authored without further clarification; user-visible strings are quoted verbatim from source. Requirements state intended behavior; a `**Status:** Not satisfied` line marks requirements the running system does not yet meet (see [known gaps](../GAPS.md)). Conventions, the ID scheme, and the acceptance-test trace-header format are defined in the [capability registry](../README.md).

---

## L2-NOTIF-047: Notification bell and unread count

**Traces to:** L1-NOTIF-011

Authenticated users shall see a notification bell with an unread badge.

**Acceptance Criteria:**

1. Given unread notifications, when the bell renders, then a badge shows the unread count, capped at "9+".
2. Given the bell is opened, when the dropdown renders, then the 20 most recent notifications are listed newest-first with relative timestamps; with none, it shows "No notifications yet".

---

## L2-NOTIF-048: Notification types and producers

**Traces to:** L1-NOTIF-011

The system shall create the four notification types at their triggering events.

**Acceptance Criteria:**

1. Given a connection request is sent, when the write completes, then the recipient receives a `connect_request` notification ("Someone wants to connect with you" / "Visit your connections to accept.").
2. Given a facilitator approves a match or a mutual connection, when the approval completes, then both participants receive a `match_approved` notification.
3. Given a chat message is sent, when the write completes, then the partner receives a `new_message` notification.
4. Given a call is scheduled, when the meeting is persisted, then the partner receives a `meeting_scheduled` notification.

---

## L2-NOTIF-049: Marking notifications read

**Traces to:** L1-NOTIF-011

Users shall mark notifications read individually or all at once.

**Acceptance Criteria:**

1. Given an unread notification, when the user clicks it, then it is marked read and the unread badge decrements.
2. Given unread notifications, when the user activates "Mark all read", then every listed notification is marked read and the badge clears.

---

## L2-NOTIF-050: Notification navigation and realtime arrival

**Traces to:** L1-NOTIF-011

Clicking a notification shall navigate to the relevant page, and new notifications shall arrive without a reload.

**Acceptance Criteria:**

1. Given a `new_message`, `connect_request`, or `meeting_scheduled` notification with a connection id, when it is clicked, then the browser navigates to `/connections/{connection_id}` (else `/connections`); a `match_approved` notification navigates to `/connections/{connection_id}` (else `/dashboard`).
2. Given the user is signed in with any page open, when a notification is inserted for them, then it appears in the dropdown and a toast with a "View" action is shown, via the realtime subscription filtered by `user_id`, without reloading.
