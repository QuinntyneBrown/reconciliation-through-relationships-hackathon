# Authentication & Access (AUTH) — L2 Detailed Requirements

Each requirement traces to exactly one L1 in [L1.md](L1.md). Acceptance criteria are written so a test can be authored without further clarification; user-visible strings are quoted verbatim from source. Requirements state intended behavior; a `**Status:** Not satisfied` line marks requirements the running system does not yet meet (see [known gaps](../GAPS.md)). Conventions, the ID scheme, and the acceptance-test trace-header format are defined in the [specification registry](../README.md).

---

## L2-AUTH-006: Account sign-up

**Traces to:** L1-AUTH-002

Visitors shall create an account at `/auth/signup` with email, password, and password confirmation, with client-side password rules.

**Acceptance Criteria:**

1. Given a password shorter than 8 characters, when the visitor submits the sign-up form, then the toast "Password must be at least 8 characters." appears and no account is created.
2. Given mismatched password and confirmation, when the visitor submits, then the toast "Passwords do not match." appears and no account is created.
3. Given valid credentials, when the visitor submits, then an account is created, a session is established, and the browser navigates to `/onboarding`.
4. Given account creation succeeds but the immediate sign-in fails, when the flow completes, then the toast "Account created — please sign in." appears and the browser navigates to `/auth/login`.
5. Given a newly created auth user, when the account is created, then a corresponding `profiles` row exists with the default role `participant`.

## L2-AUTH-007: Sign-in with journey-stage routing

**Traces to:** L1-AUTH-002

Users shall sign in at `/auth/login` with email and password and be routed to the stage of the journey they have reached.

**Acceptance Criteria:**

1. Given a facilitator's credentials, when they sign in, then the browser navigates to `/facilitator`.
2. Given a participant who has not completed onboarding, when they sign in, then the browser navigates to `/onboarding`.
3. Given a participant who completed onboarding but not learning, when they sign in, then the browser navigates to `/learn`.
4. Given a participant who completed onboarding and learning, when they sign in, then the browser navigates to `/dashboard`.

## L2-AUTH-008: Invalid credentials are retryable

**Traces to:** L1-AUTH-002

A failed sign-in shall show the backend error and leave the form usable for another attempt.

**Acceptance Criteria:**

1. Given invalid credentials, when the user submits the sign-in form, then the toast "Invalid login credentials" appears and the browser stays on `/auth/login`.
2. Given the failed attempt, when the toast has appeared, then the sign-in button is re-enabled so the user can retry.

## L2-AUTH-009: Auth callback session exchange

**Traces to:** L1-AUTH-002

`GET /auth/callback` shall exchange the provided code for a session and route the user by role and journey stage; failures shall return to sign-in with an error indicator.

**Acceptance Criteria:**

1. Given a valid `code` query parameter, when `/auth/callback` is requested, then the code is exchanged for a session and the user is redirected using the same role/stage rules as sign-in (L2-AUTH-007).
2. Given an invalid or missing code, when `/auth/callback` is requested, then the user is redirected to `/auth/login?error=auth_failed`.

## L2-AUTH-010: Anonymous access redirects to sign-in

**Traces to:** L1-AUTH-003

Unauthenticated visitors shall be redirected to `/auth/login` from any protected route.

**Acceptance Criteria:**

1. Given no session, when a visitor requests `/dashboard`, then they are redirected to `/auth/login`.
2. Given no session, when a visitor requests `/`, `/auth/login`, `/auth/signup`, or `/auth/callback`, then the page is served without redirect (public routes).

---

## L2-AUTH-011: Participant journey gating

**Traces to:** L1-AUTH-003

A signed-in participant shall be held at the earliest incomplete journey stage, enforced consistently at the session proxy, sign-in, and auth callback.

**Acceptance Criteria:**

1. Given a participant with `onboarding_completed = false`, when they request any protected route other than `/onboarding`, then they are redirected to `/onboarding`.
2. Given a participant with `onboarding_completed = true` and `learning_completed = false`, when they request any protected route other than `/learn`, then they are redirected to `/learn`.
3. Given a participant who has completed both stages, when they request `/learn`, then it is served without redirect; when they request `/dashboard`, then it is served.

---

## L2-AUTH-012: Role separation

**Traces to:** L1-AUTH-003

Participants shall not reach facilitator routes, and facilitators shall not reach participant journey routes.

**Acceptance Criteria:**

1. Given a signed-in participant, when they request any `/facilitator` route, then they are redirected to `/dashboard`.
2. Given a signed-in facilitator, when they request `/onboarding` or `/learn`, then they are redirected to `/facilitator`.
