# Connections & Messaging (CONN) — L2 Detailed Requirements

Each requirement traces to exactly one L1 in [L1.md](L1.md). Acceptance criteria are written so a test can be authored without further clarification; user-visible strings are quoted verbatim from source. Requirements state intended behavior; a `**Status:** Not satisfied` line marks requirements the running system does not yet meet (see [known gaps](../GAPS.md)). Conventions, the ID scheme, and the acceptance-test trace-header format are defined in the [specification registry](../README.md).

---

## L2-CONN-035: Peer connection requests

**Traces to:** L1-CONN-009

Participants shall send connection requests to any eligible participant without a facilitator-created match.

**Acceptance Criteria:**

1. Given two participants with no match and no connection, when one sends a request, then the button flips to "Request pending" without a reload, the toast confirms the request, and the persisted row has the sender as `participant_a` with `participant_a_connected = true`, the recipient as `participant_b`, `status = 'pending'`, and `match_id` null.
2. Given the request, when the sender opens `/connections`, then the recipient appears with a "Pending" badge.

---

## L2-CONN-036: Cancelling a sent request

**Traces to:** L1-CONN-009

The sender shall be able to withdraw a pending request.

**Acceptance Criteria:**

1. Given a sent pending request, when the sender cancels it, then the connection row is deleted, a cancellation toast appears, and the profile action returns to "Connect with {name}".
2. Given the database policy, when a delete is attempted, then it succeeds only for a participant of that connection and only while `status = 'pending'`.

---

## L2-CONN-037: Mutual consent activates or queues the connection

**Traces to:** L1-CONN-009

A connection shall activate only when both parties consent; when auto-matching is disabled, mutual consent shall queue the connection for facilitator review instead.

**Acceptance Criteria:**

1. Given a pending connection where the partner has not yet consented, when the participant consents, then their `*_connected` flag is set, the toast "Connection request sent!" appears, and the chat remains locked.
2. Given the partner already consented and `system_settings.auto_matching_enabled` is `true`, when the participant consents, then the connection becomes `status = 'active'` with `connected_at` set and chat unlocks.
3. Given the partner already consented and `auto_matching_enabled` is `false`, when the participant consents, then the connection becomes `status = 'pending_review'`, the toast "Both connected! A facilitator will review your match shortly." appears, and the chat shows "Under facilitator review".
4. Given a connection that is not active, when the chat page renders, then the composer is disabled with the placeholder "Connect first to start chatting" and the connect prompt explains the current state ("RTR has matched you with {partner}!" with "Both parties must connect to activate chat.", or "Waiting for {partner} to accept your connection request…").

## L2-CONN-038: Failed connection writes never report success

**Traces to:** L1-CONN-009

A rejected or failed connection write shall surface an error and never a false success.

**Acceptance Criteria:**

1. Given the `connections` insert fails, when a participant sends a request, then an error toast appears (e.g., "Couldn't send your connection request. Please try again." or "Failed to connect. Please try again.") and no "Connection request sent" success toast is shown.

---

## L2-CONN-039: Connections list

**Traces to:** L1-CONN-009

`/connections` shall list the participant's connections newest-first with status, and explain the empty state.

**Acceptance Criteria:**

1. Given connections, when the list renders, then each row shows the partner's name and city with an "Active" or "Pending" badge, and opening a row navigates to `/connections/{id}`.
2. Given no connections, when the list renders, then "No active connections yet" appears with an "Explore recommendations" action that navigates to `/dashboard`.

---

## L2-CONN-040: Chat access is limited to the connection's participants

**Traces to:** L1-CONN-009

Only the two participants of a connection shall open its conversation.

**Acceptance Criteria:**

1. Given a signed-in participant who is `participant_a` or `participant_b`, when they request `/connections/{id}`, then the conversation loads with both profiles, message history in order, and meetings.
2. Given any other signed-in participant, when they request that URL, then they are redirected to `/connections`.
3. Given a non-existent connection id, when it is requested, then the page responds not-found.

---

## L2-CONN-041: Sending messages

**Traces to:** L1-CONN-009

Participants in an active connection shall exchange persisted text messages, notifying the partner.

**Acceptance Criteria:**

1. Given an active connection, when a participant sends "  Thank you…  " (with surrounding whitespace), then the message is persisted trimmed, appears in the conversation, and the input clears.
2. Given a sent message, when it is persisted, then a `new_message` notification is created for the partner with the message body truncated to 80 characters.
3. Given an empty or whitespace-only input, when send is attempted, then nothing is sent.
4. Given the message insert fails, when a participant sends, then the toast "Failed to send message." appears and the typed text is restored to the input.
5. Given a newly active connection with no messages, when the chat renders, then it shows "You're connected! Say hello to {partner}."

---

## L2-CONN-042: Realtime message delivery

**Traces to:** L1-CONN-009

New messages shall appear in an open conversation without a page reload.

**Acceptance Criteria:**

1. Given both participants have the conversation open, when one sends a message, then it appears in the other's conversation via the realtime subscription (messages table INSERT filtered by `connection_id`) without reloading, de-duplicated by message id.

---

## L2-CONN-043: Upcoming meeting banner

**Traces to:** L1-CONN-009

An active conversation shall surface the next scheduled call.

**Acceptance Criteria:**

1. Given a meeting whose end time (start + duration) is in the future, when the conversation renders, then a banner shows the formatted meeting time and a "Join Zoom" link when a join URL exists.
2. Given an upcoming meeting, when the header renders, then the "Schedule call" action is hidden until that meeting has ended.
