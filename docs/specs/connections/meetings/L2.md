# Meeting Scheduling (MEET) — L2 Detailed Requirements

Each requirement traces to exactly one L1 in [L1.md](L1.md). Acceptance criteria are written so a test can be authored without further clarification; user-visible strings are quoted verbatim from source. Requirements state intended behavior; a `**Status:** Not satisfied` line marks requirements the running system does not yet meet (see [known gaps](../../GAPS.md)). Conventions, the ID scheme, and the acceptance-test trace-header format are defined in the [specification registry](../../README.md).

## L2-MEET-044: Schedule-call dialog

**Traces to:** L1-MEET-010

Participants in an active connection shall schedule a call through a validated dialog.

**Acceptance Criteria:**

1. Given an active connection with no upcoming meeting, when the participant opens "Schedule call", then a dialog offers a topic prefilled "RTR Connection Call — {me} & {partner}", a date input that disallows past dates, a time input, and a duration selector (30 minutes, 1 hour, 1.5 hours, 2 hours; default 1 hour).
2. Given a missing date or time, when the participant submits, then the toast "Please select a date and time." appears and no request is sent.
3. Given Cancel, when it is activated, then the dialog closes without side effects.
4. Given a successful response, when scheduling completes, then the toast "Meeting scheduled! Both participants have been notified." appears and the meeting appears in the conversation with a "Join Zoom" link.
5. Given a failed response, when scheduling completes, then the server-provided error (or "Failed to schedule meeting. Please try again.") is shown as a toast and the dialog remains usable for retry.

## L2-MEET-045: Meeting-creation API success contract

**Traces to:** L1-MEET-010

`POST /api/zoom/create-meeting` shall create a Zoom meeting server-side, persist it, and notify the partner.

**Acceptance Criteria:**

1. Given an authenticated participant of the connection and a body with `connectionId`, `scheduledAt`, `durationMinutes`, and `topic`, when the endpoint is called, then a Zoom meeting is created via server-to-server OAuth (scheduled type, America/Toronto timezone), a `meetings` row is inserted with the Zoom id and join/start URLs, and the response returns the meeting row.
2. Given the meeting is persisted, when the request completes, then a `meeting_scheduled` notification titled "A call has been scheduled" (body = topic) is created for the other participant.

## L2-MEET-046: Meeting-creation API error contract

**Traces to:** L1-MEET-010

The meeting API shall reject unauthorized, invalid, and failed requests with distinct statuses.

**Acceptance Criteria:**

1. Given no authenticated user, when the endpoint is called, then it responds `401` with `{"error":"Unauthorized"}`.
2. Given a missing required field, when the endpoint is called, then it responds `400` with `{"error":"Missing required fields"}`.
3. Given an authenticated user who is not a participant of the connection, when the endpoint is called, then it responds `403` with `{"error":"Forbidden"}`.
4. Given Zoom rejects the meeting creation, when the endpoint is called, then it responds `502` with `{"error":"Failed to create Zoom meeting"}` and the caller can retry (a subsequent success completes normally).
5. Given the database insert fails after Zoom succeeds, when the endpoint is called, then it responds `500` with `{"error":"Failed to save meeting"}`; any unexpected error responds `500` with `{"error":"Zoom integration error"}`.
