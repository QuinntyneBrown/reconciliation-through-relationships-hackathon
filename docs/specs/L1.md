# L1 - High-Level Requirements

Reverse-engineered from the implemented system on 2026-07-12. Sources of truth, in priority order: the acceptance test suites (`boundary-interface-tests/specs/`, `e2e/tests/`), the application source (`src/`) and database migrations (`supabase/migrations/`), the UI state catalogs (`docs/mocks/`), and the intent documents (`docs/challenge.txt`, `docs/plan.md`, `docs/DECISIONS.md`).

## Conventions

- Requirements use **must/shall** language. Detailed requirements live in [L2.md](L2.md); each L2 traces to exactly one L1.
- **Overlap rule:** a behavior that manifests at one route belongs to that feature's L1, even when it is security- or accessibility-flavored. Cross-cutting L1s (L1-015 through L1-018) own only data-layer constraints, app-wide UI invariants, and viewport patterns.
- Requirements state **intended** behavior as evidenced by code, tests, and documentation. Where the running system does not yet meet an intended requirement, the L2 carries a `**Status:** Not satisfied` line referencing the Known Gaps appendix in L2.md. No performance or observability L1 exists because the codebase states no measurable target, and reverse-engineering must not invent one.
- The platform brand is always "Reconciliation Through Relationships" (never "RTR Portal").

## L1-001: Public landing page

Anonymous visitors shall learn the program's purpose, four-step journey, and participant testimonials, and shall be routed to account creation or sign-in.

## L1-002: Authentication and session management

Users shall create accounts and sign in with email and password; sessions shall be established via the auth callback exchange and shall be terminable via sign-out.

## L1-003: Journey and role-based access control

Every route shall be gated by authentication state, role (participant or facilitator), and journey stage (`onboarding_completed`, `learning_completed`), so users can only reach the stage of the journey they have unlocked.

## L1-004: Portal navigation and account menu

Authenticated users shall navigate between their role's destinations via a primary navigation with current-page indication, and shall reach their own profile and sign out via an account menu.

## L1-005: Onboarding intake

Participants shall complete a validated five-step intake wizard that persists their profile to the `profiles` table and unlocks the learning journey.

## L1-006: Learning journey

Participants shall complete audience-appropriate learning modules with tracked progress; completing all required modules shall unlock the dashboard.

## L1-007: Dashboard recommendations and discovery

Participants shall see scored Indigenous↔non-Indigenous match recommendations, waitlist and cohort-forming states, and a searchable, filterable directory of all eligible participants with an optional consent-respecting map view.

## L1-008: Participant profiles

Participants shall view each other's profiles (facilitator profiles excluded), reach their own profile via a shortcut, and initiate or manage a connection from a profile.

## L1-009: Connections and messaging

Participants shall request, cancel, accept, and conduct one-to-one connections with realtime chat, gated by mutual consent and, when auto-matching is disabled, by facilitator review.

## L1-010: Video call scheduling

Participants in an active connection shall schedule Zoom calls through a validated dialog backed by a server-side API that creates the meeting, persists it, and notifies the partner.

## L1-011: In-app notifications

Users shall receive realtime in-app notifications for connection, match, message, and meeting events, mark them read, and navigate from a notification to the relevant page.

## L1-012: Regional map and cohorts

Users shall see aggregate regional participation counts limited to eligible, consenting participants, with regions flagged when enough participants exist to form a cohort.

## L1-013: Facilitator administration

Facilitators shall monitor participant progress, create and approve or reject matches and mutual connection requests, and configure platform settings.

## L1-014: Participant intake API

The system shall expose a schema-validated `GET`/`POST /api/participants` endpoint backed by the repository seam. (Reference seam: not used by the live onboarding UI.)

## L1-015: Security and data protection

All data access shall be protected by row-level security and server-side authorization; secrets shall remain server-side; user-provided content shall be sanitized before being rendered as HTML.

## L1-016: Data integrity and persistence

The database schema shall enforce the application's status vocabularies via constraints, maintain rows via triggers, and publish realtime changes for every table the UI subscribes to.

## L1-017: Accessibility

Interactive flows shall be operable with assistive technology: form errors shall be programmatically associated and focus-managed, and text shall meet contrast requirements.

## L1-018: Responsive design

All pages shall function across viewport sizes from extra-small (<576px) through extra-large (≥1200px), with a mobile navigation pattern on narrow screens.

## Out of scope

The following capabilities are documented in `docs/` (plan, brain-dump, architecture) but are **not implemented**; they are excluded from this specification and carry no requirement IDs:

- Magic-link (passwordless) authentication — the built system uses email + password.
- An `admin` role — the database and application recognize only `participant` and `facilitator`.
- Email reminders (Resend integration).
- Events management.
- Cohort creation and membership management UI (the `cohorts`/`cohort_members` tables exist; no creation flow does).
- Native Land Digital API integration (an external help link only).
