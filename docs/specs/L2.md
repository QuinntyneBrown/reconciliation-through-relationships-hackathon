# L2 - Detailed Requirements

Each requirement traces to exactly one L1 in [L1.md](L1.md). Acceptance criteria are written so a test can be authored without further clarification; user-visible strings are quoted verbatim from source. Requirements state intended behavior; a `**Status:** Not satisfied` line marks requirements the running system does not yet meet (see the Known Gaps appendix).

Acceptance tests must carry a trace header, e.g.:

```typescript
// Acceptance Test
// Traces to: L2-015, L2-016
// Description: Onboarding step validation and persistence
```

---

## L2-001: Landing page brand identity

**Traces to:** L1-001

The landing page shall present the platform under the brand "Reconciliation Through Relationships" with the organization logo.

**Acceptance Criteria:**

1. Given an anonymous visitor, when they load `/`, then the page displays the name "Reconciliation Through Relationships" and a visible organization logo image.
2. Given the landing page, when its text is inspected, then the retired name "RTR Portal" does not appear anywhere.

---

## L2-002: Hero invitation and tagline

**Traces to:** L1-001

The landing page hero shall present the invitation headline and the tagline "An invitation, in response to the TRC's calls to action", visible at both narrow and wide viewports.

**Acceptance Criteria:**

1. Given an anonymous visitor, when they load `/`, then the hero displays "Reconciliation begins with a relationship."
2. Given a 320px-wide viewport, when the landing page renders, then the tagline "An invitation, in response to the TRC's calls to action" is fully within the viewport.
3. Given a 1280px-wide viewport, when the landing page renders, then the same tagline is fully within the viewport.

## L2-003: Four-step journey explanation

**Traces to:** L1-001

The landing page shall explain the participant journey as four steps and shall state that participants choose whether a facilitator matches them.

**Acceptance Criteria:**

1. Given the landing page, when the journey section renders, then the four steps "Create your profile", "Complete the learning journey", "Meet your match", and "Build the relationship" are all visible.
2. Given the landing page, when the journey copy is read, then it explains that the participant can choose whether a facilitator matches them.
3. Given the landing page, when the visitor activates "See how it works", then the `#how` journey section scrolls into the viewport.

**Traces to:** L1-001

Landing page calls to action shall route visitors to sign-up and sign-in.

**Acceptance Criteria:**

1. Given the landing page, when the visitor activates "Begin your journey", then the browser navigates to `/auth/signup`.
2. Given the landing page, when the visitor activates "Sign in", then the browser navigates to `/auth/login`.

## L2-006: Account sign-up

**Traces to:** L1-002

Visitors shall create an account at `/auth/signup` with email, password, and password confirmation, with client-side password rules.

**Acceptance Criteria:**

1. Given a password shorter than 8 characters, when the visitor submits the sign-up form, then the toast "Password must be at least 8 characters." appears and no account is created.
2. Given mismatched password and confirmation, when the visitor submits, then the toast "Passwords do not match." appears and no account is created.
3. Given valid credentials, when the visitor submits, then an account is created, a session is established, and the browser navigates to `/onboarding`.
4. Given account creation succeeds but the immediate sign-in fails, when the flow completes, then the toast "Account created — please sign in." appears and the browser navigates to `/auth/login`.
5. Given a newly created auth user, when the account is created, then a corresponding `profiles` row exists with the default role `participant`.

## L2-007: Sign-in with journey-stage routing

**Traces to:** L1-002

Users shall sign in at `/auth/login` with email and password and be routed to the stage of the journey they have reached.

**Acceptance Criteria:**

1. Given a facilitator's credentials, when they sign in, then the browser navigates to `/facilitator`.
2. Given a participant who has not completed onboarding, when they sign in, then the browser navigates to `/onboarding`.
3. Given a participant who completed onboarding but not learning, when they sign in, then the browser navigates to `/learn`.
4. Given a participant who completed onboarding and learning, when they sign in, then the browser navigates to `/dashboard`.

## L2-008: Invalid credentials are retryable

**Traces to:** L1-002

A failed sign-in shall show the backend error and leave the form usable for another attempt.

**Acceptance Criteria:**

1. Given invalid credentials, when the user submits the sign-in form, then the toast "Invalid login credentials" appears and the browser stays on `/auth/login`.
2. Given the failed attempt, when the toast has appeared, then the sign-in button is re-enabled so the user can retry.

## L2-009: Auth callback session exchange

**Traces to:** L1-002

`GET /auth/callback` shall exchange the provided code for a session and route the user by role and journey stage; failures shall return to sign-in with an error indicator.

**Acceptance Criteria:**

1. Given a valid `code` query parameter, when `/auth/callback` is requested, then the code is exchanged for a session and the user is redirected using the same role/stage rules as sign-in (L2-007).
2. Given an invalid or missing code, when `/auth/callback` is requested, then the user is redirected to `/auth/login?error=auth_failed`.

## L2-010: Anonymous access redirects to sign-in

**Traces to:** L1-003

Unauthenticated visitors shall be redirected to `/auth/login` from any protected route.

**Acceptance Criteria:**

1. Given no session, when a visitor requests `/dashboard`, then they are redirected to `/auth/login`.
2. Given no session, when a visitor requests `/`, `/auth/login`, `/auth/signup`, or `/auth/callback`, then the page is served without redirect (public routes).

## L2-011: Participant journey gating

**Traces to:** L1-003

A signed-in participant shall be held at the earliest incomplete journey stage, enforced consistently at the session proxy, sign-in, and auth callback.

**Acceptance Criteria:**

1. Given a participant with `onboarding_completed = false`, when they request any protected route other than `/onboarding`, then they are redirected to `/onboarding`.
2. Given a participant with `onboarding_completed = true` and `learning_completed = false`, when they request any protected route other than `/learn`, then they are redirected to `/learn`.
3. Given a participant who has completed both stages, when they request `/learn`, then it is served without redirect; when they request `/dashboard`, then it is served.

## L2-012: Role separation

**Traces to:** L1-003

Participants shall not reach facilitator routes, and facilitators shall not reach participant journey routes.

**Acceptance Criteria:**

1. Given a signed-in participant, when they request any `/facilitator` route, then they are redirected to `/dashboard`.
2. Given a signed-in facilitator, when they request `/onboarding` or `/learn`, then they are redirected to `/facilitator`.

## L2-015: Five-step intake wizard with persistence

**Traces to:** L1-005

Onboarding shall guide the participant through five steps — Basic Info, Location & Treaty, Faith & Interests, Availability & Format, Matching Preferences — persist the profile, and open the learning journey.

**Acceptance Criteria:**

1. Given a participant on `/onboarding`, when each step renders, then a progress indicator shows "Step N of 5" for the current step.
2. Given completed earlier steps, when the participant navigates Back, then previously entered values (text fields, checked options) are preserved.
3. Given all steps valid, when the participant activates "Complete profile", then the `profiles` row is updated with every collected field, `onboarding_completed` is set to `true`, the toast "Profile saved! Let's start your learning journey." appears, and the browser navigates to `/learn`.
4. Given interests entered as tags, when they are added (via Enter or the "Add" button), then they are stored lowercased and de-duplicated, and each tag can be removed before submission.
5. Given no authenticated session at submit time, when the participant submits, then the toast "Session expired. Please sign in again." appears and the browser navigates to `/auth/login`.

## L2-024: Matching algorithm

**Traces to:** L1-007

Match recommendations shall pair only Indigenous with non-Indigenous participants and score each pair out of 100 using fixed weights.

**Acceptance Criteria:**

1. Given a participant, when candidates are scored, then only candidates whose `is_indigenous` differs from the participant's are considered.
2. Given a candidate pair, when the score is computed, then it is the sum of: location (same city case-insensitively = 30, else same province = 15, else 0), availability overlap (up to 20, from day and time overlap ratios ×10 each, rounded), shared interests (shared ÷ smaller list × 20, rounded), shared language (any = 10), same faith tradition (10), and shared participation format (any = 10).
3. Given scored candidates, when the dashboard selects recommendations, then it keeps candidates that are in an approved match or score at least 40, sorted descending, and shows at most 5.

## L2-034: Profile connect action state machine

**Traces to:** L1-008

The profile page's connect action shall reflect the current connection state and drive its transitions.

**Acceptance Criteria:**

1. Given no connection with the profile's owner, when the viewer activates "Connect with {name}", then a `connections` row is created (`participant_a` = viewer, connected; `participant_b` = owner, not connected; no match required), a `connect_request` notification is created for the owner, and the toast "Connection request sent to {name}!" appears.

## L2-037: Mutual consent activates or queues the connection

**Traces to:** L1-009

A connection shall activate only when both parties consent; when auto-matching is disabled, mutual consent shall queue the connection for facilitator review instead.

**Acceptance Criteria:**

1. Given a pending connection where the partner has not yet consented, when the participant consents, then their `*_connected` flag is set, the toast "Connection request sent!" appears, and the chat remains locked.
2. Given the partner already consented and `system_settings.auto_matching_enabled` is `true`, when the participant consents, then the connection becomes `status = 'active'` with `connected_at` set and chat unlocks.
3. Given the partner already consented and `auto_matching_enabled` is `false`, when the participant consents, then the connection becomes `status = 'pending_review'`, the toast "Both connected! A facilitator will review your match shortly." appears, and the chat shows "Under facilitator review".
4. Given a connection that is not active, when the chat page renders, then the composer is disabled with the placeholder "Connect first to start chatting" and the connect prompt explains the current state ("RTR has matched you with {partner}!" with "Both parties must connect to activate chat.", or "Waiting for {partner} to accept your connection request…").

## L2-046: Meeting-creation API error contract

**Traces to:** L1-010

The meeting API shall reject unauthorized, invalid, and failed requests with distinct statuses.

**Acceptance Criteria:**

1. Given no authenticated user, when the endpoint is called, then it responds `401` with `{"error":"Unauthorized"}`.
2. Given a missing required field, when the endpoint is called, then it responds `400` with `{"error":"Missing required fields"}`.
3. Given an authenticated user who is not a participant of the connection, when the endpoint is called, then it responds `403` with `{"error":"Forbidden"}`.

5. Given the database insert fails after Zoom succeeds, when the endpoint is called, then it responds `500` with `{"error":"Failed to save meeting"}`; any unexpected error responds `500` with `{"error":"Zoom integration error"}`.
