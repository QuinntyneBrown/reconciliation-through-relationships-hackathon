# Facilitator Administration (FACIL) — L2 Detailed Requirements

Each requirement traces to exactly one L1 in [L1.md](L1.md). Acceptance criteria are written so a test can be authored without further clarification; user-visible strings are quoted verbatim from source. Requirements state intended behavior; a `**Status:** Not satisfied` line marks requirements the running system does not yet meet (see [known gaps](../GAPS.md)). Conventions, the ID scheme, and the acceptance-test trace-header format are defined in the [specification registry](../README.md).

---

## L2-FACIL-054: Facilitator overview

**Traces to:** L1-FACIL-013

`/facilitator` shall summarize the community with five statistics and quick actions.

**Acceptance Criteria:**

1. Given a signed-in facilitator, when the overview loads, then it shows counts for "Total participants", "Completed learning", "Pending matches", "Active connections", and "Regional cohorts".
2. Given the quick actions, when one is activated, then it navigates to matching, participants, `/map`, or settings respectively.

## L2-FACIL-055: Participants directory (facilitator)

**Traces to:** L1-FACIL-013

Facilitators shall browse all participants with journey status and filters.

**Acceptance Criteria:**

1. Given the directory, when it renders, then each row shows name (linking to `/profile/{id}`), location, background badge, journey badge, and join date, paginated at 10 per page.
2. Given journey status, when it is derived, then `learning_completed` → "Ready"; `onboarding_completed` without learning → "Learning"; otherwise → "Onboarding".
3. Given search (name/city/province), background, and journey-status filters, when combined, then results satisfy all of them; with no results, "No participants match your filters." is shown with a "Clear filters" action.

## L2-FACIL-056: Manual match creation

**Traces to:** L1-FACIL-013

Facilitators shall create a match between one Indigenous and one non-Indigenous eligible participant, persisting the computed score.

**Acceptance Criteria:**

1. Given the matching page, when the facilitator selects one Indigenous and one non-Indigenous participant (both onboarding- and learning-complete) and activates "Create match", then a `matches` row is inserted with status `suggested`, `auto_generated = false`, the facilitator as creator, and the score/criteria computed by the matching algorithm (L2-MATENG-024); the toast "Manual match created." appears.
2. Given the insert fails, when creation is attempted, then the toast "Failed to create match." appears.

## L2-FACIL-057: Approving and rejecting matches and mutual requests

**Traces to:** L1-FACIL-013

Facilitators shall approve or reject both matches and mutual connection requests, with participants notified on approval.

**Acceptance Criteria:**

1. Given a match, when the facilitator approves it, then `matches.status` becomes `approved` with approver and timestamp, a `connections` row is created for the pair, both participants receive `match_approved` notifications, and the toast "Match approved and participants notified." appears.
2. Given a match, when the facilitator rejects it, then its status becomes `rejected`, the toast "Match rejected." appears, and it is listed under the Rejected tab.
3. Given a mutual connection request (`status = 'pending_review'`), when the facilitator approves it, then the connection becomes `active` with `connected_at` set, both participants receive `match_approved` notifications, and the toast "Connection approved." appears; rejecting deletes the connection with the toast "Connection rejected."

## L2-FACIL-058: Auto-matching toggle

**Traces to:** L1-FACIL-013

Facilitators shall control whether mutual connections activate automatically or await review, via the `auto_matching_enabled` setting. (See GAP-008 for the contradiction with the "approval is always required" settings copy.)

**Acceptance Criteria:**

1. Given the matching page or settings page, when the facilitator toggles auto-matching, then `system_settings.auto_matching_enabled` is updated.
2. Given the setting is `false`, when two participants mutually consent, then the connection awaits facilitator review (L2-CONN-037 criterion 3); given `true`, it activates immediately (L2-CONN-037 criterion 2).

---

## L2-FACIL-059: Settings persistence

**Traces to:** L1-FACIL-013

Facilitator settings shall persist the auto-matching switch and cohort threshold, and report failures.

**Acceptance Criteria:**

1. Given the settings page, when the facilitator saves, then `auto_matching_enabled` and `cohort_threshold` are upserted to `system_settings` with the updater and timestamp, and the toast "Settings saved." appears; the values survive a reload.
2. Given the cohort threshold input, when it renders, then it accepts values from 3 to 20 (default 5).
3. Given the write fails, when the facilitator saves, then the toast "Failed to save settings." appears.

## L2-FACIL-060: Cohort threshold governs cohort displays

**Traces to:** L1-FACIL-013
**Status:** Not satisfied — the saved `cohort_threshold` is never read back; cohort logic hardcodes 5 (GAP-006).

The saved `cohort_threshold` shall drive every cohort-readiness computation (dashboard banner, regional map).

**Acceptance Criteria:**

1. Given `cohort_threshold` saved as 7, when the dashboard cohort banner and `/map` readiness are computed, then 7 — not 5 — is the minimum participant count.
